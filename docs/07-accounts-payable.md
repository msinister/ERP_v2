# 07 — Module 6: Accounts Payable

## Architecture: PO ↔ Receipt ↔ Bill (many-to-many)

**Three independent records, linked many-to-many.** This is the key architectural decision (see Module 3 for full explanation).

- **PO** = what was ordered
- **Receipt** = what physically arrived in one shipment (could span multiple POs)
- **Bill** = what the vendor invoiced for (could span multiple POs and/or receipts)

A single vendor invoice covering items from 3 separate POs creates one Bill linked to 3 POs and 1 Receipt.

## Bill entry

### Sources
- **Manual entry** (primary) — staff types in from vendor's invoice
- **Nice-to-have v2:** PDF/Excel upload with OCR + LLM extraction → drafts a bill or PO
- CSV batch import (deferred)

### Numbering
- **Bill #** = auto-generated by ERP (e.g., `BILL-00001`)
- **Vendor invoice #** = stored as `vendor_reference` field

### Bill creation flow
1. PO created → cost locked
2. Vendor ships, sends invoice
3. Receiving creates check-in list → physical receipt → confirms received → inventory updates
4. Receipt **auto-creates a draft bill** (qty + cost from receipt)
5. AP staff cross-references vendor's actual invoice → adjusts if needed → confirms bill
6. AP balance posts at this point

### Bill statuses
`Draft → Confirmed → Shipped/In Transit → Received/Closed → Cancelled`

(Aligned with PO status flow rather than typical bill statuses.)

**Payment status** is a separate flag: `Unpaid → Partially Paid → Paid`.

### Bill line fields
- Photo, title, SKU, manufacturer part number
- Qty, cost, total
- Last cost (reference)
- WAC (reference)
- Qty on hand / available / on order (reference, contextual)
- Notes
- Optional: freight per line, customs/duty per line

### Non-PO bills (expenses)
- Separate workflow from product bills
- Categorized: utility, rent, shipping expense, travel, office, professional services, etc.
- Categories self-serve in Admin Settings (add/edit/disable)
- GL account assignment per line
- Payroll deferred to its own module
- Basic expense logging in v1; full expense module deferred

### Recurring bills
**Skipped for v1.**

## Payment workflow

### Simple logging only
- No bank API integration
- No check printing
- No ACH initiation
- AP team handles actual payment outside the system, then logs it

### Record Payment
Button on bill → fields:
- Date
- Amount
- Payment method
- Account/source
- Reference number (check #, wire ref, etc.)
- Notes

System updates bill payment status (unpaid → partial → paid).

### Bill payment dashboard
- Filterable list of open bills
- Columns: vendor, bill #, vendor invoice #, bill date, due date, balance, original amount, amount paid, credits applied, status
- Sortable (default by due date)
- Export to CSV
- Multi-select for batch operations

### Overpayment
- Warning shown
- Allowed with override
- Excess becomes vendor credit on that vendor's account

### Approval thresholds
**None.** Logging-only workflow doesn't need approval gates.

### Vendor credits

**Workflow:**
- Statuses: `Draft → Confirmed → Cancelled`
- Confirmed credits assumed approved (we send vendor a copy as confirmation)
- Visible on vendor record / dashboard

**Application:**
- **Manual** — credit is treated as available balance toward that vendor
- Apply to any bill of choice, not auto-applied

### Payment void / cancel
- Manual reversal with reason note
- Bill returns to unpaid status
- Logged in audit trail

## Drop-ship vendor AR (cross-reference)

Drop-ship vendors don't appear in AP. They live in AR via the **shadow customer** model. See Module 3 for full architecture.

For mixed vendors (rare):
- Vendor record handles AP for stock products
- Shadow customer record handles AR for drop-ship commissions
- No netting

## Landed cost handling on bills

### Most vendors quote landed cost
No allocation needed at entry; cost goes directly into FIFO layer.

### Some vendors quote ex-works
Freight bill arrives separately, often weeks later, often spanning multiple POs/shipments.

### Late freight/customs bill workflow
1. Freight bill arrives → entered as a bill
2. Linked to receipt(s) (one or many)
3. **Allocated across all units in shipment** (by chosen method: unit count, weight, value, box count)
4. **Back-allocates** to FIFO layers — updates layer cost retroactively
5. Generates **COGS adjustment** for already-sold units (retroactive to original sale date — see Module 7)

## Architectural notes

### Auto journal posting on AP events
See Module 7. Summary:

| Event | JE |
|-------|-----|
| Goods received (no bill yet) | DR Inventory / CR Accrued Receipts |
| Bill confirmed (matches receipt) | DR Accrued Receipts / CR AP - Vendor |
| Bill paid | DR AP / CR Cash/Bank |
| Vendor credit issued | DR Vendor Credits (1410, asset) / CR Purchase Returns & Allowances (5150) |
| Vendor credit applied to bill | DR AP / CR Vendor Credits (1410) — AP reduced when the credit is consumed |
| Vendor credit application reversed | DR Vendor Credits (1410) / CR AP (mirror of apply) |
| Overpayment on bill payment | Payment JE DR AP / CR Cash (full); then DR Vendor Credits (1410) / CR AP for the excess |
| Late landed cost arrives | DR Inventory (active layers) / CR AP-Freight + COGS adjustment for sold units |

### "Accrued Receipts" account (clearing)
This is the holding account for "received not invoiced." It should always net to zero once everything is billed. A non-zero balance signals:
- Goods received but never billed (vendor never invoiced you)
- Bill posted but receipt never confirmed (data error)

Periodic reconciliation report should surface aged Accrued Receipts balances.
